Agent Teams And Skills

The working model is team-oriented and stage-specialized: scanner, analyzer, architect, generator, verification, repair, quality, and push teams each own a skill domain with clear input and output contracts.

Open Agent Team Model

How The Pipeline Works

The canonical implementation flow is Scanner → Analyzer → Architect → CodeGen → Verifier → Quality Gates → Pusher, with checkpoint durability, run diagnostics, and optional repair loops between generation and release.

Open Pipeline Overview
End-to-End Flow

The orchestrator runs each repository through the same stateful pipeline. Each stage updates a RepoEvolutionState, writes checkpoints, emits run events, and can resume from saved work.

1
Input
A batch of RepoManifest objects or one repo arrives from the CLI or API. The manifest carries name, URL, category, stars, metadata, and optional commit SHA.
pipeline/__main__.py
pipeline/api/server.py
2
Clone
The source repo is cloned or fetched from GitHub API, filtered into a bounded file set, and optionally served from a commit-aware cache when the same SHA was already processed.

If source files exist, the LLM performs deep analysis. Otherwise, the system falls back to a design brief generated from repo metadata only.
pipeline/agents/analyzer.py

Exact Runtime Working

How a Batch Run Actually Starts

The CLI or API constructs manifests, then creates a PipelineOrchestrator. For batch runs, the orchestrator sorts manifests by stars and last-updated time, applies a concurrency semaphore, and processes the input in batches.

python -m pipeline run --manifests data/manifests.example.json
POST /run with manifests, optional batch size, optional concurrency, optional enabled stages

How a Single Run Actually Starts

The system maps the string category to RepoCategory, builds a one-repo manifest, allocates a run id, and passes that into the orchestrator. Single runs now participate in the diagnostics system because they emit a real run id.

What the State Contains

Each repo has a RepoEvolutionState carrying source files, deep analysis or design brief, mobile architecture, generated files, generation result, verification report, warnings, errors, timing, correlation id, checkpoint schema version, and stage history.

How Resume Works Now

The system first tries to load a checkpoint from the data directory. If the checkpoint is already completed, the repo is skipped and the monitor is updated. Otherwise, resume logic prefers explicit stage-history records; only when those records do not exist does it fall back to heuristic field-based skips.

How Reliability Was Strengthened

Checkpoints are written atomically using temp file replacement. Each checkpoint is schema-versioned. Clone work is cached per commit SHA. Pushes use an idempotency key derived from manifest, architecture, and generated content so duplicate reruns do not republish the same result.

How Events and Diagnostics Work

Each stage emits run-aware events that include run id, correlation id, repo, stage, attempt, and duration when available. The diagnostics endpoint combines event history, status, and final report for a run into one response.

GET /report/{run_id}
GET /diagnostics/{run_id}
GET /status
GET /metrics
Stage-by-Stage Working

Cloning

The scanner attempts a shallow git clone first. If clone fails, it falls back to GitHub REST API tree/content fetches. Files are filtered by extension, max size, max file count, and total character budget. Priority is given to package/config/readme files before broader source traversal.

  • Commit-aware clone cache only activates when commit_sha exists.
  • Assets are inlined as base64 data URLs for analysis.
  • Output becomes state.source_files.

Analyzing

The analyzer builds a condensed source summary and asks the LLM for framework, routes, components, state model, API calls, design theme, readiness score, and mobile translation notes. If no source files were extracted, it creates a metadata-driven design brief instead.

  • Deep path: analyze_repo_deep()
  • Fallback path: analyze_repo_shallow()
  • Output becomes state.deep_analysis or state.design_brief.

Architecting

The architect converts analysis into a mobile execution plan. It chooses Expo or React Native, sets navigation type, constructs screens, assigns state management, chooses the navigation library, and derives dependency lists.

  • High complexity webapps can switch to React Native.
  • Screen list may come from routes, design brief, or fallback heuristics.
  • Output becomes state.architecture.

Generating

The code generator creates a mobile file map in memory. It emits package configuration, TypeScript config, Expo config, navigation files, screens, shared components, theme files, API service, state store, README, ARCHITECTURE doc, and CI config.

  • It uses a project context manager to supply source-aware context to code generation.
  • It creates both machine-facing files and human-facing documentation.
  • Output becomes state.generated_files.

Verifying

The verifier writes the generated file map into a temp project directory and runs actual package-manager and build commands. It captures stdout, stderr, exit code, duration, failure category, summary, and relevant file hints.

  • Checks include install, lint, type-check, test, and optional web export.
  • Failed reports can keep the temp directory in non-strict mode.
  • Output becomes state.verification_report.

Repairing

If verification fails and repair is enabled, the repair agent asks the LLM for a minimal edit plan. The prompt now includes failure category, summary, and relevant files so repairs are narrower and less likely to rewrite unrelated screens.

  • Repair applies create/update/delete edits to the in-memory file map.
  • The verifier is then re-run up to the configured repair attempt limit.
  • This is the generate → verify → repair loop implemented in the repo today.

Quality Check

Quality gates inspect the generated output for structure, config validity, minimum screens, navigation presence, CI file, README, architecture docs, secret leaks, typing, accessibility, offline support, error handling, placeholder content, design tokens, SCA, dependency freshness, performance hints, and i18n readiness.

  • The quality score is computed from passed and warning gates.
  • Generation result stores files, lines, build status, gates, and score.
  • Output becomes state.generation_result.

Pushing

If verification policy allows, the pusher creates or finds a GitLab project, writes CI config, commits generated files in batches, configures project settings, and appends an audit log entry. If an identical push already exists, it reuses the previous GitLab URL.

  • Pushes are blocked when configured to stop on verification failure.
  • Project name is derived from source repo name and framework.
  • Audit log goes to data/logs/push_audit.jsonl.

Implemented Platform Features

Per-Run Isolation

The API no longer mutates global config for every request. Batch size, concurrency, enabled stages, and run id are passed as per-run overrides into the orchestrator.

Stage Journal

Every repo records stage started, completed, and failed records with attempt and duration data, which makes resume behavior explicit instead of inferred.

Atomic Checkpointing

Checkpoint writes use temp-file replacement and include schema versioning for safer persistence and future migrations.

Idempotent Side Effects

Cloning uses commit-aware caching and pushing uses content-based deduplication backed by the audit log.

Structured Failure Taxonomy

Verifier failures now classify into categories like dependencies, lint, types, tests, build, timeout, and infrastructure.

Targeted Repair

Repair prompts now receive failure category, summary, and relevant-file hints so repairs are scoped instead of broad and generic.

Diagnostics Endpoint

A single diagnostics request can now return the final report, active status, and run-filtered event timeline.

Monitoring and Cost Summary

The monitor tracks progress, stage times, quality scores, categories, and estimated LLM cost per run.

Current Implementation Notes

Actual Mobile Targets

The current generator and architecture model are centered on Expo and React Native. Flutter and Ionic appear in older mappings or comments, but the active implementation path is Expo / React Native.

Verification Loop Is Real

This is not a fake quality estimate. The project writes generated files to disk and runs real package-manager and build-related commands during verification.

Version Drift Exists

Some public strings still say version 2.0.0 while other config and newer behavior align more closely with a newer internal state of the project.

CLI and API Are Not Identical

The API path has had more recent run-isolation improvements. The CLI still contains older configuration mutation behavior in its command wrapper code.

Bottom Line What this project actually does today is: accept repo manifests, orchestrate a durable per-repo state machine, analyze source or metadata, design a mobile architecture, generate a mobile app codebase, verify it with real commands, repair it when verification fails, score it with quality gates, and optionally publish the result to GitLab with deduplicated side effects and observable run diagnostics.
